# Requirements: Tender Panel

**Spec:** 009-tender-panel
**Status:** draft
**Date:** 2026-04-07

---

## Feature Overview

Панель управления тендерами — REST API + domain layer для просмотра списка тендеров,
которые были спарсены и отправлены по почте. Пользователи могут фильтровать и сортировать
тендеры, а также отмечать тендер как "взято в работу".

Текущая аудитория: администраторы. Архитектура должна быть готова к появлению
нескольких пользователей с разными ролями в будущем (без реализации ролей сейчас).

---

## User Stories

### US-1: Просмотр списка тендеров
**Как** пользователь панели,
**Я хочу** видеть постраничный список всех тендеров,
**Чтобы** понимать что у нас в работе и что было отправлено.

**Критерии приёмки:**
- Список возвращается постранично (default: 20, max: 100)
- Каждый тендер показывает все ключевые поля
- Результаты отсортированы по дате создания (desc) по умолчанию

### US-2: Фильтрация тендеров
**Как** пользователь панели,
**Я хочу** фильтровать тендеры по разным параметрам,
**Чтобы** быстро находить нужные тендеры.

**Критерии приёмки:**
- Фильтр по региону (точное совпадение)
- Фильтр по заказчику (частичное совпадение, case-insensitive)
- Фильтр по статусу (один или несколько)
- Фильтр по полю `takenInWork` (true/false/all)
- Фильтры комбинируются (AND логика)

### US-3: Сортировка тендеров
**Как** пользователь панели,
**Я хочу** сортировать список по разным полям,
**Чтобы** видеть наиболее актуальные или важные тендеры первыми.

**Критерии приёмки:**
- Сортировка по: `createdAt`, `deadline`, `amount`, `customer`, `region`
- Направление: `asc` / `desc`
- Только одна сортировка за раз

### US-4: Отметить тендер "взято в работу"
**Как** пользователь панели,
**Я хочу** отмечать тендер как взятый в работу,
**Чтобы** команда знала, что по нему идёт работа.

**Критерии приёмки:**
- PATCH `/api/v1/tenders/{id}` с `{ "takenInWork": true/false }`
- Возвращает обновлённый тендер
- Остальные поля не обновляются через этот endpoint

## Functional Requirements

### P0 — обязательно к реализации

| ID | Требование |
|----|-----------|
| F-01 | Сущность `Tender` с полями: id, number, title, region, customer, amount, status, deadline, takenInWork, createdAt, updatedAt |
| F-02 | `GET /api/v1/tenders` — постраничный список с фильтрами и сортировкой |
| F-03 | `GET /api/v1/tenders/{id}` — получить тендер по ID |
| F-04 | `PATCH /api/v1/tenders/{id}` — обновить поле `takenInWork` |
| F-05 | Flyway миграция для таблицы `tenders` |
| F-06 | Защита эндпоинтов через существующую аутентификацию |

### P1 — важно

| ID | Требование |
|----|-----------|
| F-07 | Enum `TenderStatus`: `SENT`, `WON`, `LOST`, `IN_PROGRESS` |
| F-08 | Фильтр по нескольким статусам одновременно (`?status=SENT,WON`) |
| F-09 | Индексы на колонки фильтрации: `region`, `customer`, `status`, `taken_in_work` |
| F-10 | Валидация входящих параметров с понятными сообщениями об ошибках через `AppException` |

### P2 — желательно

| ID | Требование |
|----|-----------|
| F-11 | Фильтр по диапазону дат (`deadlineFrom`, `deadlineTo`) |
| F-12 | Фильтр по диапазону суммы (`amountFrom`, `amountTo`) |
| F-13 | Поиск по номеру тендера (`numberSearch`, partial match) |

## Non-Functional Requirements

| Категория | Требование |
|-----------|-----------|
| Производительность | Запрос списка (без сложных фильтров) ≤ 200ms при 10k записей |
| Безопасность | Все эндпоинты требуют аутентификации |
| Расширяемость | Структура готова к добавлению поля `assignedUserId` без миграции логики |
| Тесты | Unit-тесты сервисного слоя, интеграционные тесты контроллера (`@WebMvcTest`) |
| Код стиль | Соответствие ktlint + detekt правилам проекта |

---

## Constraints & Assumptions

- Тендеры **создаются только парсером** (внешняя система), не через этот API
- Тендеры **не привязаны к пользователю** (поле `assignedUserId` — будущее)
- Аутентификация **уже реализована** — переиспользуем существующий механизм
- База данных — PostgreSQL 17, ORM — Spring Data JPA

---

## Out of Scope

- Создание / удаление тендеров через API
- Обновление полей кроме `takenInWork`
- Уведомления при изменении статуса
- Роли пользователей (admin vs regular)
- Экспорт в Excel / CSV
